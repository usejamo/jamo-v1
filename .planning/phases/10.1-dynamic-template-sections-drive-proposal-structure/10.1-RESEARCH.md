### Locked Decisions

- **D-01:** Template sections define WHICH sections exist ‚Äî architectural shift from slot-driven (9 hardcoded positions) to section-driven (derived from selected template, identified by UUID). A 12-section template produces a 12-section proposal.

## Summary

Phase 10.1 replaces the hardcoded 9-section model with a template-driven architecture. The central challenge is a coordinated refactor spanning: (1) a DB migration to extend `proposal_sections` with `name`, `description`, `position`, `role` columns and add `is_default` to `templates`; (2) deletion of `SECTION_WAVE_MAP`/`SECTION_NAMES` constants and the wave orchestration loop; (3) a new sequential generation engine that reads `proposal_sections` rows by position; (4) an edge function payload upgrade to v2; and (5) workspace components (`SectionWorkspace`, `SectionNavPanel`) transitioning from hardcoded key-iteration to DB-driven section arrays.

The biggest risk is that existing proposals have `proposal_sections` rows with `section_key` (a role string) but no `name`, `description`, or `position` ‚Äî they must be backfilled in the migration or handled gracefully in display code. The second risk is the tight coupling between `SectionWorkspace`/`SectionNavPanel` and `SECTION_WAVE_MAP` from `generation.ts` ‚Äî both files import and iterate `Object.keys(SECTION_WAVE_MAP)` directly, meaning the component refactor and the constant deletion must happen atomically.

**Primary recommendation:** Deliver in three atomic sub-tasks: (1) migration + seed, (2) generation engine rewrite, (3) workspace/UI wiring. Each sub-task must leave the app in a runnable state.

### `templates` table ‚Äî missing column [VERIFIED: migration 20260418000023]

| Column | Type | Purpose |
|--------|------|---------|
| `is_default` | boolean NOT NULL DEFAULT false | Identifies the Standard Proposal default template |

The templates table currently has no `is_default` column. The 3 pre-built templates exist but none is flagged as default.

### Unique constraint issue

The current UNIQUE constraint is `(proposal_id, section_key)`. This worked when `section_key` was a role string that could never repeat. With dynamic templates, two sections could theoretically have the same role (e.g., two `scope_of_work` sections in a template). The planner should consider:

**Option A:** Keep constraint as-is (sufficient for Standard Proposal; non-standard templates shouldn't have duplicate roles anyway ‚Äî acceptable for 10.1 scope).

**Option B:** Drop `UNIQUE(proposal_id, section_key)` and add `UNIQUE(proposal_id, position)` instead ‚Äî cleaner for the new model.

**Recommendation (ASSUMED):** Option B is cleaner long-term. The planner should confirm. Either way, the edge function's `writeSection()` must stop using `onConflict: 'proposal_id,section_key'` and write by `id` directly.

### New sequential generation loop in `useProposalGeneration.ts`

**Fetch sections from DB (replaces `createInitialSections`):**
```typescript
const { data: sections } = await supabase
  .from('proposal_sections')
  .select('id, name, description, role, position, status, content')
  .eq('proposal_id', proposalId)
  .order('position', { ascending: true })
```

**Sequential loop (replaces three-wave logic):**
```typescript
const completedSections: Array<{ id: string; name: string; content: string }> = []

for (const section of sections) {
  const ragChunks = await fetchRagChunks(orgId, section.name, therapeuticArea)
  
  const payload: GenerateSectionPayloadV2 = {
    version: 2,
    proposalId,
    sectionId: section.id,
    sectionName: section.name,
    sectionDescription: section.description,
    sectionRole: section.role,
    priorSections: completedSections,
    proposalContext: { studyInfo, assumptions, services },
    ragChunks,
    tone: state.tone,
    debug,
  }
  
  const content = await streamSection(section.id, payload)
  if (content) {
    completedSections.push({ id: section.id, name: section.name, content })
  }
}
```

- **Option A (recommended for 10.1):** Retain `consistencyAnchor` in payload, continue extracting anchor after section 1 (understanding), pass to all subsequent sections. `priorSections` array provides additional coherence. No separate anchor extraction pass needed after that.
- **Option B:** Remove anchor entirely; rely solely on `priorSections` array. Simpler but riskier for coherence.

The existing `extractAnchor()` function and anchor edge function branch can remain unchanged in 10.1.

### `buildSectionPrompt` update

The function currently uses `SECTION_NAMES[sectionId]` as the section name (a lookup by role). In v2:
- `sectionName` is passed directly ‚Äî no lookup needed
- `sectionDescription` is appended to the section-specific instruction: `Generate the "${sectionName}" section. ${sectionDescription ? `The section covers: ${sectionDescription}` : ''}`
- `sectionRole` selects prompt strategy hints (tone emphasis, structure notes) ‚Äî role-specific hints added as an addendum, not a replacement
- `priorSections` injected as: `[PRIOR SECTIONS]\n${priorSections.map(s => `## ${s.name}\n${s.content}`).join('\n\n')}\n[/PRIOR SECTIONS]`
- `templateContext` block removed (now expressed via section rows, not as separate injection)

### `writeSection` update

## 6. Supabase Realtime

### Current pattern [VERIFIED: useProposalGeneration.ts lines ~265-285]

```typescript
const channel = supabase
  .channel(`proposal-sections-${proposalId}`)
  .on('postgres_changes', {
    event: '*',
    schema: 'public',
    table: 'proposal_sections',
    filter: `proposal_id=eq.${proposalId}`,
  }, (payload) => {
    const section = payload.new
```

**Established Supabase Realtime pattern from Phase 7 (current code):** Already working correctly for the `proposal_sections` table ‚Äî the channel setup, filter syntax, and cleanup pattern are battle-tested. The only change is the key field.

## 8. Architecture Patterns

### Pattern 1: Pre-created sections (D-04)

Sections are created at wizard `handleGenerate`, before navigation. This is the "optimistic nav" pattern ‚Äî user arrives at workspace and sees the full section list immediately.

```typescript
// In handleGenerate(), after proposal created:
const { data: templateSections } = await supabase
  .from('template_sections')
  .select('name, description, role, position')
  .eq('template_id', selectedTemplateId)
  .order('position', { ascending: true })

const sectionRows = templateSections.map(ts => ({
  proposal_id: proposalId,
  org_id: orgProfile.org_id,
  section_key: ts.role ?? `section_${ts.position}`,  // fallback for null-role sections
  section_name: ts.name,
  name: ts.name,
  description: ts.description,
  role: ts.role,
  position: ts.position,
  status: 'pending',
  content: '',
}))

await supabase.from('proposal_sections').insert(sectionRows)
```

**Note on `section_key` for null-role sections:** The unique constraint is currently on `(proposal_id, section_key)`. For null-role sections, a fallback like `section_${position}` avoids constraint violations. If the unique constraint is changed to `(proposal_id, position)` (Option B), `section_key` can be set to the UUID or left as the role string.

### Pattern 2: DB-driven workspace init

`SectionWorkspace` currently fills gaps using `Object.keys(SECTION_WAVE_MAP)` for keys not in the sections prop. In 10.1, the sections prop IS the complete set (pre-created from template). The fill-gaps loop is removed.

```typescript
// NEW: sections prop is complete, just map it
const sectionsMap: Record<string, SectionEditorState> = {}
for (const s of sections) {
  sectionsMap[s.id] = { ...s, autosave_status: 'idle', ai_action: null }
}
dispatch({ type: 'SET_SECTIONS', payload: sectionsMap })
dispatch({ type: 'SET_ACTIVE_SECTION', payload: sections[0]?.id ?? null })
```

### Pattern 3: Sequential generation with prior context

Each section receives the content of all previously completed sections. The `priorSections` array grows as generation progresses. For a 12-section template, section 12 receives up to 11 prior sections' content. Token budget consideration: planner may choose to summarize or truncate prior sections if payload size becomes a concern. The edge function's existing `max_tokens: 1500` limit for generation is unchanged.

## 9. Risks and Gotchas

### Risk 1: Existing proposals break after migration

**Problem:** Existing `proposal_sections` rows have `position = NULL` (column doesn't exist yet) and `section_key` = role string. After migration adds `position` and the workspace switches to position-based ordering, existing proposals must have `position` backfilled correctly.

**Mitigation:** The migration includes a `CASE` backfill for the 9 standard roles. Proposals generated before Phase 10.1 will continue to display correctly.

**Residual risk:** Existing proposals will NOT have a `selected_template_id` on the `proposals` row (it was optional in Phase 10 per D-02 of that phase). After 10.1 makes template mandatory, the proposals query for template badge will return NULL. Display must handle `NULL` gracefully: "Template: Standard Proposal" (infer from section_key pattern) or just omit the badge for pre-10.1 proposals.

### Risk 2: `SectionWorkspace` + `SectionNavPanel` import `SECTION_WAVE_MAP` directly

Both files call `Object.keys(SECTION_WAVE_MAP)` to derive the ordered list of sections. Deleting `SECTION_WAVE_MAP` without updating these files simultaneously will break the workspace entirely. These three changes must be atomic in a single plan task:
1. Remove `SECTION_WAVE_MAP` from `generation.ts`
2. Update `SectionWorkspace` to use sections prop
3. Update `SectionNavPanel` to use sections prop

### Risk 3: `useAutosave` writes by `section_key`

**Must verify:** If `useAutosave` constructs its Supabase UPDATE with `.eq('section_key', sectionKey)`, it will fail when `sectionKey` is a UUID ‚Äî because the `section_key` column still holds role strings for existing rows. The fix is to change the autosave write to `.eq('id', sectionKey)` (where `sectionKey` in the new model is actually the section UUID).

**Check:** `src/hooks/useAutosave.ts` ‚Äî must confirm how it identifies the row to update.

### Risk 4: Edge function writes by `section_key` unique constraint

Current `writeSection()` uses `onConflict: 'proposal_id,section_key'`. In v2, sections are pre-created and must be updated by `id`. The upsert must change to a targeted UPDATE. If the unique constraint is also changed (Option B), the old upsert would fail anyway.

### Risk 5: Test file `useProposalGeneration.test.ts` imports deleted exports

The test file imports `createInitialSections`, `getWaveSections`, `TOTAL_SECTIONS` from `generation.ts`. These must be removed from both the types file AND the test file. The test file will need significant rewriting since `makeInitialState()` calls `createInitialSections()`.

### Risk 6: Template badge requires `templateName` in workspace

`ProposalDetail.tsx` must pass `templateName` to `SectionWorkspace`. This requires either:
- Joining `proposals ‚Üí templates(name)` in the ProposalDetail fetch
- Or a separate query for the template name after proposal load

**Recommendation:** Add `template:templates(name)` to the proposal fetch in `ProposalsContext` or `ProposalDetail`.

### Risk 7: Pre-selection of default template at Step 4

`Step4Generate.tsx` and `WizardState` must handle default template pre-selection. The wizard must query `templates WHERE is_default = true` on mount to seed `state.selectedTemplateId`. If the query fails or returns nothing, generation must still work (fall back gracefully ‚Äî but "no template selected" is no longer a valid state per D-16, so failure here is a hard error).

### Risk 8: `section_key` column and the unique constraint coexistence

For new proposals (created after 10.1), `section_key` will be set to either the `role` value (if non-null) or a generated fallback. For sections with null role, duplicate `section_key` values within a proposal are possible (e.g., two custom sections with `section_key = 'section_3'`). This is a strong argument for changing the unique constraint from `(proposal_id, section_key)` to `(proposal_id, position)` ‚Äî it avoids the fallback naming problem entirely.
