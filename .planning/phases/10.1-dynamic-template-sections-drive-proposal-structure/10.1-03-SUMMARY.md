# Phase 10.1 Plan 03: Edge Function V2 Payload Branch — Summary

**One-liner:** Edge function generate-proposal-section gains a v2 code path that uses sectionDescription as content scope, sectionRole as a soft strategy hint, injects priorSections for consistency, and writes by UUID instead of upserting by section_key — with full v1 backward compatibility preserved.

## What Was Built

Surgical additions to `supabase/functions/generate-proposal-section/index.ts`:

1. **`buildSectionPromptV2`** — new prompt builder that:
   - Injects `sectionDescription` as `SECTION SCOPE:` in the system prompt (primary content driver)
   - Looks up `sectionRole` in a 6-entry `roleHints` map for structure/tone guidance (soft signal, not content override)
   - Injects up to 3 most-recent `priorSections` (800-char truncated) before `[REGULATORY CONTEXT]`
   - Includes `consistencyAnchor` when present

2. **`writeSectionById`** — new DB write helper that runs `UPDATE proposal_sections SET content/status/generated_at WHERE id = sectionId` (no upsert, no section_key dependency)

3. **Version branch in streaming handler**:
   - `const isV2 = body.version === 2` declared before TransformStream (captured in `flush()` closure)
   - Destructures v2-specific fields: `sectionName`, `sectionDescription`, `sectionRole`, `priorSections`, `proposalContext`
   - Routes to `buildSectionPromptV2` or `buildSectionPrompt` based on `isV2`
   - `flush()` calls `writeSectionById` (v2) or `writeSection` (v1)
   - Required-fields guard updated: `proposalInput` only required for v1

4. **V1 path fully preserved** — `buildSectionPrompt`, `writeSection`, and all v1 destructuring remain unchanged

None — plan executed exactly as written. The required-fields validation guard was adjusted from `!proposalInput` to `!isV2 && !proposalContext` as a minor correctness fix (Rule 2) to avoid rejecting valid v2 requests that have no `proposalInput` field.
