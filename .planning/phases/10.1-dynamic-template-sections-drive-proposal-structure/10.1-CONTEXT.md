# Phase 10.1: Dynamic Template Sections Drive Proposal Structure - Context

**Gathered:** 2026-04-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the hardcoded 9-section model with a template-driven section architecture. When a proposal is created, its sections are instantiated from the selected template — not from hardcoded constants. The standard 9 sections become a "default template" (a real DB row), not a special code case. Generation, workspace display, and ProposalsList all become section-data-driven.

This phase does NOT include editing or swapping templates from the workspace (deferred to Phase 10.2). It does NOT reintroduce wave-based parallelism. It does NOT build a template editor UI.

</domain>

<decisions>
## Implementation Decisions

### Structural Scope — What "Drive Structure" Means

- **D-01:** Template sections define WHICH sections exist in a proposal — not just their labels. The architectural shift is from **slot-driven** (9 hardcoded positions with fixed prompts) to **section-driven** (sections derived from the selected template, identified by UUID). A 12-section oncology template produces a 12-section proposal. A 7-section bioequivalence template produces a 7-section proposal.

- **D-02:** The standard 9 sections become a **real `templates` table row** with `is_default: true`, seeded via migration. "No template selected" is not a valid or user-facing state. The wizard always has a template pre-selected (default is the pre-selection). All templates are treated equally by the data model — no hardcoded special cases. This template should be named something like "Standard Proposal" or equivalent.

### Identifier Architecture

- **D-03:** Three separate identifiers, each with a distinct role:
  - **UUID (row primary key)** — stable identifier for `proposal_sections` rows. Used for routing, FK references, React keys, any place code needs to reference a section. Generated at insert, never changes.
  - **`role`** — optional generation hint (nullable `text`). Known roles (`scope_of_work`, `understanding`, etc.) let the edge function reach for a role-specific prompt strategy. `null` is a valid first-class choice — falls back to generic prompt. Never use role as a primary identifier.
  - **`name`** — display only. Can change freely. Never use as an identifier.

### Section Instantiation

- **D-04:** `proposal_sections` rows are created **at wizard completion** (when the user clicks Generate in Step 4), before generation starts. Rows are copied from the selected template's `template_sections`, preserving `name`, `description`, `role`, and `position`. All rows start with `status = 'pending'`. The user sees the full section structure immediately in the workspace — this is the payoff for pre-creation.

- **D-05:** The workspace **does not look up the template at render time**. The template was the source at creation; after that, the proposal owns its sections. No template FK lookup needed for display.

### Role Classification for Uploaded Templates

- **D-06:** LLM-classify `role` at **template extraction time** (inside the `template-extract` edge function), not during proposal generation. The resolved role is stored on `template_sections.role` and copied to `proposal_sections.role` at instantiation. User can confirm/edit the pre-filled classification in the template review UI (surface LLM's guess pre-filled, editable, `null` / "None" must be a selectable option). Pre-built templates have roles set explicitly in seed data — no LLM pass needed.

### Generation Engine

- **D-07:** Generation is **sequential by `position`** — the wave system is retired. Generation walks `proposal_sections` rows ordered by `position`, one at a time. Each section can include prior completed sections' content in its prompt (prior context passed in payload or fetched). Resume/retry: a failed section retries in place, then generation continues from the next position. The default template's seed data must encode position such that executive_summary and cover_letter are last (they depend on all other sections being complete first).

- **D-08:** `SECTION_WAVE_MAP`, `SECTION_NAMES`, `createInitialSections()`, and `getWaveSections()` in `src/types/generation.ts` are **deleted**. Any code still referencing them after migration is a bug. The `wave` field on `proposal_sections` can be left as dead metadata or removed (planner decides — either is acceptable). No "legacy path" preserved — all proposals go through the dynamic section model.

### Edge Function Payload Contract (v2)

- **D-09:** New payload shape for `generate-proposal-section`. A `version: 2` field is added for future compatibility. **Same code path for all sections** — the standard 9 are now just template sections that happen to always have a known role. Do not keep a parallel "v1-style" code path:

```typescript
interface GenerateSectionPayloadV2 {
  version: 2
  proposalId: string
  sectionId: string              // proposal_sections UUID
  sectionName: string            // from template section name
  sectionDescription: string | null  // from template section — critical for uploaded templates
  sectionRole: string | null     // soft signal for prompt strategy selection
  priorSections: Array<{         // already-completed sections in position order
    id: string
    name: string
    content: string
  }>
  proposalContext: {             // unchanged from current
    studyInfo: { ... }
    assumptions: Array<{ ... }>
    services: string[]
  }
  ragChunks: Array<{ content: string; doc_type: string; agency: string }>
  tone: ToneOption
  debug?: boolean
}
```

  - `role` selects a **prompt strategy** (tone, structure hints, emphasis) — not the content scope. An uploaded "Scope of Work" section classified as `scope_of_work` still generates from its `sectionName` + `sectionDescription`; role contributes hints, not overrides.
  - `sectionDescription` is the primary content scope for uploaded template sections. Template extraction must capture it from document headings/subheadings. The template review UI must allow editing.
  - `templateContext` block (injected in Phase 10) is **removed** from the payload — template is expressed via section rows, not as a separate injection block.
  - Remove `consistencyAnchor` from payload if prior section context now serves the same coherence purpose (planner decides — may retain both initially).

### Workspace Display

- **D-10:** `proposal_sections` is the **single source of truth** for workspace display. `ProposalDetail` and `SectionWorkspace` fetch sections by `proposal_id`, ordered by `position`. Nav and editor consume from the same store/context. No hardcoded fallbacks.

- **D-11:** Supabase Realtime subscription on `proposal_sections` filtered by `proposal_id`. Workspace renders the full section nav immediately (all `status: 'pending'`), updating each section as generation completes. Sequential generation streams section completions to the UI progressively — the user sees progress.

- **D-12:** Queued (pending) sections show as a section nav entry + empty editor placeholder with a "Not yet generated" state. The nav is fully populated before any generation starts.

### Section Nav Template Badge

- **D-13:** A small muted label/badge appears **above the section list** in the workspace nav showing the template name (e.g., "Template: Standard Proposal" or "Template: Phase I/II Oncology"). Applies to all proposals including the default — the UI treats all templates identically. The badge is **interactive** — clicking it opens a stub panel in Phase 10.1 (full edit/swap UI deferred to Phase 10.2). Template name in badge reflects creation-time template name; does not update if sections are later edited.

- **D-14:** Uploaded templates must have a name at upload time (force name input at upload, or auto-generate from filename if blank). "Template: Untitled" must never appear.

### ProposalsList

- **D-15:** Show the template name on proposal cards in `ProposalsList`. Requires `proposals.selected_template_id` (already in `database.types.ts` per Phase 10 verification). Join on `templates` for name at query time. Small, in scope for Phase 10.1.

### Wizard Step 4 Update

- **D-16:** The wizard's "No template selected" state no longer exists. The default template is pre-selected when the user reaches Step 4. The template selector shows all available templates with the default pre-highlighted. The ContextSummary always shows a template name.

### Claude's Discretion
- Whether `consistencyAnchor` is retained alongside prior section context or removed (may retain both initially as a safety net)
- Whether the `wave` field is dropped via migration or left as dead metadata
- How much prior section content to include in the payload (all prior, or summarized if token budget requires)
- Exact prompt construction inside the edge function for role-specific vs. generic sections
- Whether template badge stub panel shows any content (empty, or a simple "Edit sections — coming soon")

</decisions>

<specifics>
## Specific Ideas

- **Default template name:** "Standard Proposal" (or similar — planner picks something clear)
- **Template badge in nav:** "Template: [name]" — small, muted, above section list. Same visual weight as a label/chip, not a header.
- **Empty state during generation:** Full nav with all sections showing pending state. Section editors show "Not yet generated" placeholder. Nav is populated before any generation starts.
- **ProposalsList template column:** Join `proposals` → `templates` on `selected_template_id`, show template name in the proposal card/row. Small change.
- **`is_default` flag:** Add `is_default boolean NOT NULL DEFAULT false` to `templates` table via migration. Seed the default template with `is_default: true`. Unique constraint or application-level enforcement to ensure only one.

</specifics>

- **Wave-based parallelism** — May reintroduce as narrow optimization on independent sections if performance warrants. Not resurrecting as default.
