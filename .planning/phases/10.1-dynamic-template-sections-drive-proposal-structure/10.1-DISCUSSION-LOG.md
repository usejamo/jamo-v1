**User's choice:** At wizard completion.

### What sections does a proposal use when no template is selected?

| Option | Description | Selected |
|--------|-------------|----------|
| Still the standard 9 (unchanged) | Non-template proposals keep hardcoded 9 sections. Two paths. | |
| Standard 9 become a built-in default template | Default template is a real DB row with is_default: true, seeded via migration. All proposals go through the dynamic-section path. | ✓ |

**User's choice:** Option 2.

**Notes:** "Make the default template a real row in your templates table, flagged as is_default: true or similar, and seed it via migration. Don't make it a magic string or a code constant that the DB references. The whole point is that it's data, not code. In the wizard UI, 'no template selected' probably shouldn't even be a user-facing concept anymore. It's 'Default' (or 'Standard Proposal') as one of the pre-built options, pre-selected."

### Section key/ID architecture

| Option | Description | Selected |
|--------|-------------|----------|
| Use role field as key | Role as primary identifier with slug fallback. | |
| Use generated slug from name | Slug-of-name for all template sections. | |
| Use position index | Sequential 1, 2, 3... | |
| Separate UUID / role / name concerns | None of the above — three identifiers for three concerns. | ✓ |

**User's choice:** Separate identifier architecture.

**Notes:** Full response: "UUID as stable identifier on proposal_sections — used for routing, foreign keys, React keys. Role as optional metadata for generation pipeline (nullable). Name as display only, never an identifier. For unrecognized sections in uploaded templates: LLM-classify at template extraction time, then let the user confirm in the template review UI. Classification happens once, during extraction — not during proposal generation. 'None / generic' must be a valid choice. Sections that genuinely don't map to a known role get a null role and fall back to a generic prompt at generation time. Pre-built templates have roles set explicitly in seed data."

## Generation Engine

### Section ordering / wave structure

| Option | Description | Selected |
|--------|-------------|----------|
| Template position order, single wave | Sequential by position. No wave grouping. | ✓ |
| Role-based wave assignment still applies | Known roles inherit standard wave. Null-role → wave 2. | |
| You decide | Leave to planner. | |

**User's choice:** Option 1 — sequential by template position, waves retired.

**Notes:** "Waves don't survive the move to template-defined sections. The wave system assumed the standard 9 with known roles and known dependencies; once templates can have arbitrary sections (many with null roles from uploads), wave assignment degrades — Option 2 ends up dumping most null-role sections into wave 2 in parallel, which is exactly what's blowing the token budget today. Sequential also unlocks prior-section context: each section can reference already-generated content in its prompt. Default template ordering: preserve current workspace nav order. Executive summary and cover letter must be last — encode this in seed data for the default template's section positions; don't rely on wave logic. Retire the wave field, or leave it as dead metadata."

### Edge function payload shape

| Option | Description | Selected |
|--------|-------------|----------|
| sectionId becomes role (or generic key) | Interface stays similar. Role-based prompt selection. | |
| New payload shape with section identity | Function receives name, description, optional role. Name+description = primary frame. Role = additive. | ✓ |
| You decide | Leave to planner. | |

**User's choice:** Option 2 — new payload shape with section identity.

**Notes:** "Option 2 inverts the relationship correctly: name and description are the primary signal of what this section is supposed to be, and role is a hint that lets you swap in a more specialized prompt strategy when available. Don't keep two edge function paths — migrate the standard 9 to flow through the new payload shape too (they just happen to always have a role set). Description matters more than you might expect — for uploaded templates especially, it's how the generator knows 'this section should cover X, Y, Z.' Role becomes a soft signal, not a contract. Worth bumping a version field on the edge function payload now."

## Workspace Display

### How does ProposalDetail/SectionWorkspace know which sections to show?

| Option | Description | Selected |
|--------|-------------|----------|
| Read proposal_sections from DB (by proposal ID) | Single source of truth. ORDER BY position. | ✓ |
| Read from template + hardcoded fallback | Two paths. | |

**User's choice:** Option 1.

**Notes:** "proposal_sections is the source of truth. Generation writes to it. The UI reads from it. One direction, one place. This means deleting any code path that reads SECTION_NAMES for display purposes. Search for it, kill all usages, then delete the constant. The workspace shouldn't need to know what template the proposal was created from — the template was the source of the sections at creation time; after that, the proposal owns its sections."

### Queued section display

| Section nav entry + empty editor placeholder | Full nav populated with pending state. | ✓ |

**User's choice:** Nav entry + empty editor placeholder.

### Section nav template badge

| Option | Description | Selected |
|--------|-------------|----------|
| Template name shown at top of nav as badge | "Template: X" above section list. | ✓ |
| No template indicator in workspace nav | Clean nav, no extra chrome. | |
| You decide | Leave to Claude. | |

**User's choice:** Show template badge.

**Notes:** "Apply to default template too — 'Template: Standard Proposal'. Templates are load-bearing now; hiding the template makes the proposal's structure feel arbitrary. Make it interactive — clicking is the natural home for 'edit sections' and 'swap template' actions. Just the name. Don't dynamically change the label if sections are later edited. Uploaded templates show whatever the user named them — force a name at upload."
