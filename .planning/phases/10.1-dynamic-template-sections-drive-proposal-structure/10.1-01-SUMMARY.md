# Phase 10.1 Plan 01: Schema Migration for Template-Driven Sections — Summary

**One-liner:** SQL migration adds name/description/position/role to proposal_sections and is_default to templates, seeds the "Standard Proposal" default template with 9 position-ordered sections, and backfills all existing proposal data.

## What Was Built

A single migration file (`20260427000024_template_driven_sections.sql`) containing 8 SQL operations executed in order:

1. `ALTER TABLE proposal_sections` — adds `name text`, `description text`, `position integer`, `role text` (all nullable, IF NOT EXISTS guards)
2. `ALTER TABLE templates` — adds `is_default boolean NOT NULL DEFAULT false`
3. `CREATE UNIQUE INDEX templates_single_default` — partial index WHERE is_default=true, enforcing only one default (T-10.1-01 mitigation)
4. `INSERT INTO templates` — "Standard Proposal" with fixed UUID `00000000-0000-0000-0000-000000000001`, `is_default=true`, `source='prebuilt'`
5. `INSERT INTO template_sections` — 9 rows for Standard Proposal with positions 1–7 (content sections) then 8 (executive_summary) and 9 (cover_letter) last per D-07
6. `UPDATE proposal_sections SET position` — backfills from section_key CASE mapping, ELSE 99 for unknowns, WHERE position IS NULL
7. `UPDATE proposal_sections SET name` — backfills display name from section_key, WHERE name IS NULL
8. `UPDATE proposals SET selected_template_id` — backfills to Standard Proposal UUID for proposals with sections but no template set

Migration applied to live Supabase project `fuuvdcvbliijffogjnwg`. TypeScript types regenerated — `database.types.ts` now includes `is_default`, `position`, `name`, `description` columns on the relevant table types.
