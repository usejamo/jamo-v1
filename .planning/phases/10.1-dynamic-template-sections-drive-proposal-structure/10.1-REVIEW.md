# Phase 10.1: Code Review Report

**Reviewed:** 2026-04-27
**Depth:** standard
**Files Reviewed:** 18
**Status:** issues_found

## Summary

This phase adds template-driven proposal section architecture: templates from the DB now drive which sections are created and their order, replacing the previous hardcoded 9-section structure. The core flow is sound and well-structured. Two critical issues were found — an unauthenticated edge function call path and an unhandled null dereference in the edge function. Six warnings cover missing error handling on Supabase writes, a race condition in section completion, and a setTimeout-based fallback that can fire stale dispatches. Four info items address code quality.

## Critical Issues

### CR-01: Edge function uses service role key without verifying user authentication

**File:** `supabase/functions/generate-proposal-section/index.ts:396-416`

**Issue:** The function creates a `userClient` to resolve the calling user, but if the `Authorization` header is missing or invalid, `userClient.auth.getUser()` returns `{ data: { user: null } }` rather than throwing. The subsequent `user!.id` non-null assertion (line 413) and `profile!.org_id` assertion (line 416) will throw unhandled runtime errors with an unhelpful 500 response — rather than a clear 401. More critically, a request with no `Authorization` header would reach the Anthropic call and write to the DB using the service role key without any authenticated user being confirmed.

**Fix:**
```typescript
const { data: { user }, error: authError } = await userClient.auth.getUser()
if (authError || !user) {
  return new Response(JSON.stringify({ error: 'Unauthorized' }), {
    status: 401,
    headers: { ...corsHeaders, 'Content-Type': 'application/json' },
  })
}

const { data: profile } = await supabase
  .from('user_profiles')
  .select('org_id')
  .eq('user_id', user.id)
  .single()

if (!profile?.org_id) {
  return new Response(JSON.stringify({ error: 'User profile not found' }), {
    status: 403,
    headers: { ...corsHeaders, 'Content-Type': 'application/json' },
  })
}
const orgId: string = profile.org_id
```

### CR-02: `proposal_sections` insert in wizard ignores `org_id` when `profile` is null

**File:** `src/components/ProposalCreationWizard.tsx:239-256`

**Issue:** `sectionInserts` uses `profile?.org_id ?? null` for `org_id`. The `proposal_sections` table has a non-nullable FK to `organizations` (`proposal_sections_org_id_fkey`). If `profile` is null at the time `handleGenerate` runs (auth context not yet resolved), every row insert will have `org_id: null`, which violates the FK constraint and silently fails (the error is only logged, not surfaced to the user — line 255). The user is then navigated to generation with no sections pre-created, causing generation to find no rows and abort.

**Fix:** Guard the entire template-section pre-creation block on `profile?.org_id`:
```typescript
if (!profile?.org_id) {
  console.error('[ProposalCreationWizard] org_id not available — cannot pre-create sections')
  // Still navigate; ProposalDetail will show a recoverable empty state
} else {
  // ... existing template fetch + sectionInserts block
  const sectionInserts = templateSections.map((ts) => ({
    ...
    org_id: profile.org_id,  // guaranteed non-null here
  }))
}
```

## Warnings

### WR-01: `setTimeout` fallback for `SECTION_COMPLETE` can fire after section already completed via Realtime

**File:** `src/hooks/useProposalGeneration.ts:367-369`

**Issue:** After streaming finishes, a 10-second `setTimeout` unconditionally dispatches `SECTION_COMPLETE` with the locally-accumulated `fullText`. If the Realtime subscription fires first (the normal path), the reducer will apply `SECTION_COMPLETE` twice: the second dispatch overwrites `finalContent` with the local SSE-accumulated value, which may differ from what was written to DB (e.g., if the DB write succeeded but accumulated text was incomplete). The `setTimeout` is never cancelled.

**Fix:** Track whether the section already completed and cancel the timeout on completion:
```typescript
let completed = false
const fallbackTimer = setTimeout(() => {
  if (!completed) {
    dispatch({ type: 'SECTION_COMPLETE', sectionId: section.id, content: fullText })
  }
}, 10000)

// In the Realtime handler (or after streamSection resolves), set completed = true and clearTimeout(fallbackTimer)
```
Alternatively, make the reducer idempotent: skip the update if `status` is already `'complete'`.

### WR-02: `TemplatesTab` polling interval leaks on unmount during the first render cycle

**File:** `src/components/settings/TemplatesTab.tsx:263-281`

**Issue:** The `useEffect` that manages the poll interval has a cleanup that calls `clearInterval(pollRef.current)` and sets it to null. However, the cleanup runs before the effect body on re-runs, and on the very first run with `hasProcessing = true`, the cleanup of the *previous* effect iteration also clears the interval just set. In practice this causes a missed poll tick on the initial processing cycle. Additionally, if the component unmounts while `hasProcessing` is still true, the cleanup correctly clears the interval — but `fetchTemplates` may still resolve and call `setTemplates` on an unmounted component.

**Fix:** Add an `isMounted` flag inside `fetchTemplates` or use an abort pattern:
```typescript
useEffect(() => {
  let cancelled = false
  async function fetch() {
    const { data } = await supabase.from('templates').select(...)
    if (!cancelled && data) setTemplates(data as Template[])
    if (!cancelled) setLoading(false)
  }
  fetch()
  return () => { cancelled = true }
}, [])
```

### WR-03: `ProposalCreationWizard` creates a duplicate draft proposal if `createProposal` call in step-1 effect and `handleGenerate` both run

**File:** `src/components/ProposalCreationWizard.tsx:141-158` and `198-215`

**Issue:** The `useEffect` on line 141 creates a draft proposal when `state.step === 1 && !state.proposalId`. `handleGenerate` (line 200) also creates a proposal if `!id`. If the step-1 effect's `.then()` hasn't resolved by the time the user clicks Generate (e.g., due to slow network), `state.proposalId` will still be null, causing two proposals to be created. Only the second one is navigated to; the first becomes an orphan.

**Fix:** Dispatch `SET_SUBMITTING` immediately in `handleGenerate` before checking `id`, and add a guard:
```typescript
async function handleGenerate() {
  if (state.submitting) return  // prevent double-click race
  dispatch({ type: 'SET_SUBMITTING', value: true })
  ...
}
```
Also consider moving proposal creation entirely to `handleGenerate` and removing the step-1 effect that eagerly creates a draft.

### WR-04: `buildSlidingWindow` in `chatContext.ts` silently truncates mid-conversation

**File:** `src/utils/chatContext.ts:65-73`

**Issue:** The sliding window breaks as soon as a single message would exceed the `budgetChars` limit, even if earlier messages fit. A single large assistant message near the end of history would cause all messages before it to be excluded, producing a chat history with only partial context. The `break` should be a `continue` (skip that message, keep trying older ones) or the function should use a different strategy.

**Fix:**
```typescript
// Replace break with continue to skip oversized individual messages
if (used + cost > budgetChars) continue
```
Note: if the intent is strictly "most recent N chars", `break` is correct but should be documented. Currently the JSDoc says "walks messages backwards, includes whole messages" implying `continue` is the intended behavior.

### WR-05: `SectionWorkspace` DB write for consistency check has no error handling

**File:** `src/components/editor/SectionWorkspace.tsx:152-158`

**Issue:** The `.then()` call that persists `consistency_flags` and `consistency_check_ran` to Supabase (lines 152-158) uses `.then()` with no error callback. A failed DB write is silently swallowed. If the write fails, the in-memory state shows the check ran but the DB row is not updated, so on the next page load the check will re-run (since `consistency_check_ran` in DB is still false).

**Fix:**
```typescript
supabase
  .from('proposals')
  .update({ consistency_flags: flags as any, consistency_check_ran: true })
  .eq('id', proposalId)
  .then(({ error }) => {
    if (error) console.error('[SectionWorkspace] Failed to persist consistency flags:', error)
  })
```

### WR-06: `generate-proposal-section` edge function writes section content before confirming stream completed successfully

**File:** `supabase/functions/generate-proposal-section/index.ts:479-487`

**Issue:** The `flush()` callback in the `TransformStream` writes `fullText` to the DB via `writeSectionById`. However, `flush()` is called when the writable side closes — which happens whether the stream completed cleanly or was aborted midway. If the client disconnects or Anthropic closes the stream early, a partial `fullText` will be written with `status: 'complete'`, making the section appear finished with truncated content. There is no check that `[DONE]` was received or that `fullText` is non-empty.

**Fix:**
```typescript
flush() {
  if (!fullText || fullText.trim().length === 0) {
    console.warn('[generate-proposal-section] flush called with empty fullText — skipping write')
    return
  }
  const writeOp = isV2
    ? writeSectionById(supabase, sectionId, fullText)
    : writeSection(supabase, proposalId, sectionId, orgId, fullText)
  writeOp.catch((err) =>
    console.error('[generate-proposal-section] flush write error:', err)
  )
}
```

### IN-01: `ProposalDetail` uses `as any` casts throughout for `proposal` fields

**File:** `src/pages/ProposalDetail.tsx:183`, `193`, `312-321`

**Issue:** Several fields on `proposal` that exist in the DB schema (`selected_template_id`, `indication`, `sponsor_name`, `countries`, `services`) are accessed via `(proposal as any).field`. These are typed fields in `database.types.ts` but the local `Proposal` type used by `ProposalsContext` doesn't include them. This masks type errors and means any renaming would not be caught at compile time.

**Fix:** Extend the `Proposal` type in `src/types/proposal.ts` to include `selected_template_id`, `indication`, `countries`, and `services`, sourced from the `Database` types.

**Issue:** The nav item label renders `editorState?.section_key ?? key`, which shows the raw key string (e.g., `scope_of_work`) rather than a human-readable name. <span className="leading-snug">

### IN-04: `parseSectionsFromText` in `template-extract` will match most short lines as section headings

**File:** `supabase/functions/template-extract/index.ts:32-57`

**Issue:** The heuristic `isShortNoPunct` matches any line under 80 characters without a trailing punctuation character. For PDFs with dense text, this will over-detect section headings (e.g., table cell content, figure captions, short paragraphs). This inflates `sections.length`, causing `isLowConfidence` to be `false` for templates that are actually low quality. The practical impact is that low-quality PDF templates are shown as "Ready" without the amber warning banner.

**Fix:** Tighten the heuristic — require at minimum that the line is not a continuation of a sentence (starts with a capital letter) and is either title-cased or all-caps:
```typescript
const isTitleCase = /^[A-Z][a-z]/.test(trimmed) && trimmed.split(' ').filter(w => /^[a-z]/.test(w)).length < trimmed.split(' ').length / 2
const isAllCaps = trimmed === trimmed.toUpperCase() && /[A-Z]/.test(trimmed)
if ((isAllCaps || isTitleCase) && trimmed.length < 80 && !/[.!?,;]$/.test(trimmed)) { ... }
```
