# Phase 999.1: Persist Compliance Flags to DB — Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Make compliance flags (yellow warning indicators, placeholder notices) survive navigation. Currently stored only in React state inside `SectionWorkspaceContext` — wiped whenever the user routes away from `ProposalDetail`. This phase adds DB persistence so flags reload when the user returns to a proposal.

Scope: read + write compliance flags to DB. No new flag rules, no new compliance UI, no cross-proposal reporting.

</domain>

<decisions>
## Implementation Decisions

### D-01: Storage Shape — JSONB column on `proposal_sections`
Add `compliance_flags JSONB DEFAULT '[]'` to the existing `proposal_sections` table via migration. Flags travel with the section row — one query covers content + flags, no joins needed.

**Rationale:** Fits the "survive navigation" scope. No cross-proposal reporting requirement exists yet. Migrate to a normalized table later if reporting becomes a real requirement (backfill from JSONB is straightforward).

**Typed schema:** Define a typed interface (or Zod schema) for the stored flag shape, even though storage is schemaless. The `ComplianceFlag` interface in `src/types/workspace.ts` is the source of truth — extend or validate against it.

### D-02: Staleness Policy — Re-run compliance on mount, DB flags as optimistic cache
On `ProposalDetail` mount:
1. Load DB flags immediately → dispatch `SET_COMPLIANCE_FLAGS` for instant display
2. Fire `checkCompliance` in the background for each section that has content
3. Replace flags in-place silently when fresh results arrive — no flash, no animation

If fresh results === stored flags, no visible change. Cost is near-zero in the happy path.

Do NOT show a stale indicator or re-check button — avoids UI complexity without meaningfully improving accuracy.

### D-03: Write Timing — Immediately on `SET_COMPLIANCE_FLAGS` dispatch
When `useComplianceCheck` dispatches `SET_COMPLIANCE_FLAGS`, upsert the `compliance_flags` JSONB column for that section row immediately. One upsert per check, on an existing `proposal_sections` row.

Do NOT piggyback autosave — compliance results arrive after the autosave cycle has already fired, so tracking "flags dirty" state adds complexity for minimal benefit.

### Claude's Discretion
- Exact Zod schema vs. TypeScript interface for flag shape validation
- Whether to load all section flags in one query on mount or per-section as sections initialize
- Error handling for the upsert (silent fail is acceptable — flags are non-blocking)

</decisions>
