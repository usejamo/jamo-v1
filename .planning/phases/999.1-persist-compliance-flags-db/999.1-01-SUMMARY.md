# Phase 999.1 Plan 01: Persist Compliance Flags to DB Summary

## What Was Built

### Task 1: Migration + ComplianceFlagDB type

Created `supabase/migrations/20260402000021_compliance_flags_jsonb.sql`:
- `ALTER TABLE proposal_sections ADD COLUMN IF NOT EXISTS compliance_flags JSONB NOT NULL DEFAULT '[]'::jsonb`
- Column comment documents the stored shape

Added `ComplianceFlagDB` type alias to `src/types/workspace.ts`:
- `export type ComplianceFlagDB = Pick<ComplianceFlag, 'id' | 'section_key' | 'type' | 'message' | 'source'>`
- Explicitly names the DB contract so future divergence from `ComplianceFlag` is safe

### Task 2: Wire DB upsert into useComplianceCheck

Added `persistFlags` helper (memoized via `useCallback`) inside `useComplianceCheck`:
- Calls `supabase.from('proposal_sections').update({ compliance_flags: ... }).eq('proposal_id', ...).eq('section_key', ...)`
- Wraps in try/catch with silent fail — compliance flags are non-blocking per D-03
- Casts flags via `as unknown as Json` — established Supabase JSONB pattern

`await persistFlags(sectionKey, flags)` added after all 4 `SET_COMPLIANCE_FLAGS` dispatch sites:
1. Rule flags found (ruleFlags.length > 0 path)
2. Haiku error fallback (error || !data path) — refactored inline array to named `errorFlags` variable
3. Haiku success path (haikuFlags dispatch)
4. Catch block fallback — refactored inline array to named `catchFlags` variable
