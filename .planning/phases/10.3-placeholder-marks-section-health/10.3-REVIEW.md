# Phase 10.3: Code Review Report

**Reviewed:** 2026-04-28T00:00:00Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

This phase introduces a placeholder mark system (TipTap `PlaceholderMark` extension), a migration path from legacy `[PLACEHOLDER: ...]` string patterns to UUID-keyed HTML spans, and section health signalling via `SectionNavPanel` status dots. The core data flow — server generates `[PLACEHOLDER: ...]` text, `flush()` converts it to spans before DB write, client loads spans and TipTap parses them via `parseHTML` — is sound.

Two critical issues were found: an authentication null-dereference crash on the server when a JWT resolves to no user, and a streaming-vs-saved content race where the raw SSE stream (containing unconverted `[PLACEHOLDER: ...]` strings) is piped to the client before `flush()` converts them, meaning the client displays and may persist the legacy format. Four warnings cover logic correctness: the initial `saveNow` call in `SectionEditorBlock` saves `migratedContent` rather than the editor's post-parse HTML (so any normalisation TipTap does is lost), issues count does not distinguish resolved vs. unresolved placeholders, the resolution plugin marks every node touched by any transaction as a placeholder candidate without verifying the node still holds a mark (causing unnecessary work), and `authHeader` is forwarded without null-checking.

## Critical Issues

### CR-01: Null dereference crash when JWT resolves to no user

**File:** `supabase/functions/generate-proposal-section/index.ts:431-439`

**Issue:** `userClient.auth.getUser()` can return `{ data: { user: null }, error }` when the JWT is invalid or expired. The code then calls `user!.id` with a non-null assertion, which throws a runtime exception, causing the entire edge function to return a 500 rather than a meaningful 401. Additionally, `profile` can be `null` if no matching `user_profiles` row exists, and `profile!.org_id` will throw similarly.

```typescript
// current (lines 429-439)
const {
  data: { user },
} = await userClient.auth.getUser()

const { data: profile } = await supabase
  .from('user_profiles')
  .select('org_id')
  .eq('user_id', user!.id)   // crashes if user === null
  .single()

const orgId: string = profile!.org_id  // crashes if profile === null
```

**Fix:**
```typescript
const {
  data: { user },
  error: userError,
} = await userClient.auth.getUser()

if (!user || userError) {
  return new Response(JSON.stringify({ error: 'Unauthorized' }), {
    status: 401,
    headers: { ...corsHeaders, 'Content-Type': 'application/json' },
  })
}

const { data: profile, error: profileError } = await supabase
  .from('user_profiles')
  .select('org_id')
  .eq('user_id', user.id)
  .single()

if (!profile || profileError) {
  return new Response(JSON.stringify({ error: 'User profile not found' }), {
    status: 403,
    headers: { ...corsHeaders, 'Content-Type': 'application/json' },
  })
}

const orgId: string = profile.org_id
```

### CR-02: Client receives raw `[PLACEHOLDER: ...]` strings over the SSE stream

**File:** `supabase/functions/generate-proposal-section/index.ts:492-518`

**Issue:** The `TransformStream` pipes raw chunks from Anthropic directly to the client (`controller.enqueue(chunk)`) while also accumulating `fullText`. The placeholder-to-span conversion happens only in `flush()`, which runs after the stream closes — at which point the converted text is written to the DB. However, the client is already receiving and rendering the raw `[PLACEHOLDER: ...]` markdown-style text during streaming.

When the client accepts the AI action (`handleAcceptAIAction` in `SectionEditorBlock.tsx` line 130), it calls `markdownToHtml(aiAction.preview_content)` on the streamed raw text. This means:
1. The DB contains correctly converted span HTML (from `flush()`).
2. The in-memory editor content (and any immediate `saveNow`) contains the pre-conversion raw text, potentially overwriting the correct DB version on the next autosave.

**Fix:** Either (a) post-process the accumulated SSE text on the client side using `migratePlaceholders` before storing it in `preview_content`, or (b) convert `[PLACEHOLDER: ...]` patterns inline as they are streamed (detecting complete tokens in the buffer). Option (a) is simpler:

In `useProposalGeneration.ts` (or wherever `UPDATE_AI_PREVIEW` is dispatched), wrap the accumulated delta:
```typescript
import { migratePlaceholders } from '../lib/migratePlaceholders'

// When building preview_content from SSE chunks:
dispatch({
  type: 'UPDATE_AI_PREVIEW',
  payload: { section_key: sectionKey, content: migratePlaceholders(accumulated) },
})
```

And in `handleAcceptAIAction` (SectionEditorBlock.tsx line 129-130), remove the redundant `markdownToHtml` call on `preview_content` since the content will already be HTML from the server.

## Warnings

### WR-01: Migration `saveNow` persists pre-TipTap HTML, not post-parse editor HTML

**File:** `src/components/editor/SectionEditorBlock.tsx:70-75`

**Issue:** When `migratedContent !== rawContent`, the code immediately calls `saveNow(migratedContent)` (the string produced by `migratePlaceholders`). However, TipTap's `useEditor` may normalise or transform this HTML during the parse step (e.g., adding implicit `<p>` wrappers, re-serialising inline elements). The editor's canonical representation is `editor.getHTML()`. Saving `migratedContent` instead of `editor.getHTML()` means the DB can diverge from what TipTap considers the current document, which could cause a content jump on the next page load.

**Fix:**
```typescript
useEffect(() => {
  if (migratedContent !== rawContent && editor) {
    // Wait for TipTap to finish its initial render, then read back the
    // canonical HTML rather than the pre-parse migrated string.
    saveNow(editor.getHTML())
  }
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, []) // run once on mount only
```

### WR-02: `resolveStatus` treats any non-empty `issues` entry as `needs-review`, even resolved/empty ones

**File:** `src/components/editor/SectionNavPanel.tsx:17-19`

**Issue:** `Object.values(editorState.issues).some(list => list.length > 0)` correctly checks whether any issue list is non-empty. However, `issues` is `Partial<Record<IssueCategory, SectionIssue[]>>` — categories that have been explicitly set to `[]` (all placeholders resolved) are still iterated. This is currently correct, but the status dot will flash `needs-review` transiently during the initial mount of `SectionEditorBlock` because `UPDATE_SECTION_ISSUES` with the full placeholder list is dispatched after a 200 ms debounce. There is no loading state, so a section with zero placeholders will briefly show `complete`, then `needs-review` once the debounce fires, then back to `complete` when the editor re-renders with the resolved list.

More critically: if `issues` contains a category key mapped to `undefined` (which `Partial<Record<...>>` permits at the TypeScript level but the code does not guard), `list.length` will throw.

**Fix:** Guard against undefined values:
```typescript
const hasIssues = Object.values(editorState.issues).some(
  (list) => Array.isArray(list) && list.length > 0
)
```

### WR-03: `authHeader` forwarded without null guard

**File:** `supabase/functions/generate-proposal-section/index.ts:426-427`

**Issue:** `req.headers.get('Authorization')` returns `string | null`. It is stored in `authHeader` and then used as `authHeader!` (non-null assertion) when constructing `userClient`. If the header is absent, this crashes with a runtime error instead of returning a 401.

**Fix:**
```typescript
const authHeader = req.headers.get('Authorization')
if (!authHeader) {
  return new Response(JSON.stringify({ error: 'Missing Authorization header' }), {
    status: 401,
    headers: { ...corsHeaders, 'Content-Type': 'application/json' },
  })
}
```

### WR-04: Resolution plugin over-eagerly marks all nodes in touched transactions as placeholder candidates

**File:** `src/components/editor/extensions/PlaceholderMark.ts:37-55`

**Issue:** The plugin's `appendTransaction` handler iterates `tr.mapping.maps` and for each step-map it scans `stepDoc.nodesBetween(0, end, ...)` adding every node that carries a placeholder mark to `touchedIds`. However, `tr.mapping.maps` has one entry per step, and `(tr as any).docs?.[i - 1]` is an internal TipTap/ProseMirror implementation detail that is not part of the public API. If the internal shape changes, `stepDoc` silently falls back to `tr.before`, causing every step to scan the same (pre-transaction) document snapshot. This means editing non-placeholder content in a document that happens to contain placeholders elsewhere will add those unrelated placeholder IDs to `touchedIds` and then remove their marks — resolving placeholders the user did not touch.

**Fix:** Scope the scan to the positions actually changed by the step. Use `step.getMap().forEach` to identify affected ranges, then only scan nodes in those ranges:
```typescript
for (const tr of transactions) {
  if (!tr.docChanged) continue
  tr.steps.forEach((step, i) => {
    const stepMap = step.getMap()
    stepMap.forEach((_oldStart, _oldEnd, newStart, newEnd) => {
      const doc = i === 0 ? tr.before : tr.docs?.[i - 1] ?? tr.before
      doc.nodesBetween(newStart, newEnd, (node: any) => {
        const mark = node.marks?.find((m: any) => m.type === markType)
        if (mark) touchedIds.add(mark.attrs.id)
      })
    })
  })
}
```

## Info

### IN-01: `escapeHtml` does not escape single quotes

**File:** `src/lib/escapeHtml.ts:1-7`

**Issue:** The function escapes `&`, `<`, `>`, and `"` but not `'` (single quote / apostrophe). In the current usage — building `data-placeholder-label="..."` attributes with double-quote delimiters — this is safe because single quotes cannot break out of double-quoted attributes. However, if a future callsite uses the output in a single-quoted attribute context, injection becomes possible. This is an `info`-level item because the current usage is safe.

**Fix:** Add single-quote escaping for defence-in-depth:
```typescript
.replace(/'/g, '&#39;')
```

### IN-02: `any` type used extensively in `PlaceholderMark.ts`

**File:** `src/components/editor/extensions/PlaceholderMark.ts:14-15, 32, 44`

**Issue:** `doc: any`, `_oldState: any`, `newState: any`, and several inline `any` casts suppress TypeScript's ability to catch shape errors against the ProseMirror API. Given that `(tr as any).docs` is already relying on an undocumented internal (see WR-04), the lack of types here increases the chance that a ProseMirror version bump breaks the plugin silently.

**Fix:** Import proper ProseMirror types:
```typescript
import type { Node as ProseMirrorNode } from '@tiptap/pm/model'
import type { EditorState, Transaction } from '@tiptap/pm/state'
```
Replace `any` with `ProseMirrorNode`, `EditorState`, and `Transaction` where applicable.

### IN-03: `buildSectionPromptV2` uses `a.value || a.content` with `any` cast on assumptions

**File:** `supabase/functions/generate-proposal-section/index.ts:266`

**Issue:** `assumptions.map((a: any) => ...)` casts each assumption to `any` to access both `a.value` and `a.content` (a compatibility shim for v1/v2 payload shape differences). This silently accepts any object shape and produces no error if both fields are absent — the formatted string would read `- [category] undefined`.

**Fix:** Define a discriminated type or explicitly check:
```typescript
const assumptionsList = assumptions
  .map((a) => `- [${a.category}] ${a.value ?? a.content ?? '[missing]'}`)
  .join('\n')
```
And remove the `: any` cast, letting TypeScript infer the type from the `proposalContext.assumptions` array type annotation.
