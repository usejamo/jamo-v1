## Summary

Reviewed `src/lib/placeholderHtml.test.ts`, which provides unit and integration tests for `escapeHtml`, `placeholderPatternToSpan`, `migratePlaceholders`, and TipTap `Editor.create` roundtrip behaviour. The overall test structure is clear and the core contracts are well-covered. Two warning-level issues were found: one assertion that is too weak to prove the intent stated in the test (resolution plugin registration check), and one test that omits the inner-text escape assertion needed to fully validate XSS prevention. Two info-level items cover a missing UUID format assertion and a misleading comment.

## Warnings

### WR-01: Resolution plugin registration test asserts nothing specific to PlaceholderMark

**File:** `src/lib/placeholderHtml.test.ts:201-203`
**Issue:** The assertion `expect(editor.view.state.plugins.length).toBeGreaterThan(0)` is trivially satisfied by `StarterKit` alone — ProseMirror core registers several plugins before `PlaceholderMark` contributes anything. If `PlaceholderMark.addProseMirrorPlugins()` were removed entirely, this test would still pass. The test provides no signal about whether the resolution plugin is actually registered.
**Fix:** Compare plugin counts with and without `PlaceholderMark`, or check for a plugin with a known key:
```ts
// Option A — count comparison
const baseEditor = new Editor({ extensions: [StarterKit], content: html })
const fullEditor = new Editor({ extensions: [StarterKit, PlaceholderMark], content: html })
expect(fullEditor.view.state.plugins.length).toBeGreaterThan(baseEditor.view.state.plugins.length)
baseEditor.destroy()

// Option B — if the plugin exposes a PluginKey, assert on it directly
// expect(editor.view.state.plugins.some(p => p.spec.key === placeholderResolutionKey)).toBe(true)
```

### WR-02: XSS-escape test omits assertion on span inner text

**File:** `src/lib/placeholderHtml.test.ts:45-48`
**Issue:** The test titled "escapes HTML in label to prevent attribute injection" asserts that the `data-placeholder-label` attribute value is escaped, but does not assert what the span's inner text contains. `placeholderPatternToSpan` places the label as visible text between the tags as well as in the attribute. If the function were to HTML-escape the attribute but embed the raw label as inner text, DOM parsing could still lead to script injection when the HTML is inserted into a document. The attribute assertion alone does not confirm full escaping.
**Fix:** Add an assertion on the inner text:
```ts
// The inner text between > and </span> must also be escaped
expect(result).toContain('>budget &lt;amount&gt; &amp; &quot;fees&quot;<')
```

## Info

### IN-01: Distinct-UUID test does not verify UUID format

**File:** `src/lib/placeholderHtml.test.ts:69-76`
**Issue:** The test confirms that two generated IDs are non-equal, but does not verify they conform to UUID v4 format. If the implementation regressed to sequential integers or short random strings, uniqueness would still hold and the test would pass.
**Fix:** Add a format assertion alongside the uniqueness check:
```ts
const uuidRegex = /^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i
ids.forEach(id => expect(id).toMatch(uuidRegex))
```

---

### IN-02: Comment on lines 79-81 overstates test scope

**File:** `src/lib/placeholderHtml.test.ts:78-86`
**Issue:** The block comment claims the test "confirms the pattern is consumed before any markdown pass" and references `markdownToHtml`. The test body calls only `migratePlaceholders` on a plain string — `markdownToHtml` is never invoked. The comment implies coverage of the markdown interaction pipeline that does not actually exist in this test, which could mislead future contributors into believing that integration is tested.
**Fix:** Either trim the comment to accurately describe what is tested (plain-string consumption by `migratePlaceholders`), or add a companion test that actually chains `markdownToHtml → migratePlaceholders` to validate the claimed interaction.
