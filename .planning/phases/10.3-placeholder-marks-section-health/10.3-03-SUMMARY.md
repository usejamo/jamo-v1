## Key Design Decision

The §9.1 spec calls for a TipTap Editor roundtrip test (load HTML → getHTML()). After checking `vitest.config.ts` (environment: `happy-dom`), the HTML contract approach was chosen instead of instantiating a full TipTap Editor. Rationale: the UUID stability contract is "the span produced by `placeholderPatternToSpan` will be parsed by TipTap's `parseHTML` selector `span[data-placeholder-id]` and rendered back with the same UUID." This is fully verified by checking the exact attribute format without needing a DOM editor instance — TipTap's `parseHTML: [{ tag: 'span[data-placeholder-id]' }]` will match any span with that attribute, and `renderHTML` passes attrs through unchanged. The idempotency test (second `migratePlaceholders` pass is a no-op) further confirms UUID stability.
